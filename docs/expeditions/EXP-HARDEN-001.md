## Purpose

Ensure Mission Studio produces a perfectly connected planning graph where every proposal references its parent by the correct identity.

---

## Motivation

Forensic analysis of EXP-FIRSTCONTACT-003 revealed that expedition proposals reference mission IDs from the world-model node identity space, while mission proposals use a different proposal identity space. This causes aggregate parent references in the event log to be invalid, breaking navigation from expeditions to missions and from objectives to expeditions.

---

## Deliverables

1. **Correct parent references**
   - Expedition proposals must reference the mission proposal ID, not the mission node ID.
   - Objective proposals must reference the expedition proposal ID, not the expedition node ID.

2. **Identity preservation**
   - Proposal IDs must remain stable from proposal generation through snapshot approval to Genesis intake.

3. **Proposal graph validation**
   - Every expedition has a resolvable mission parent.
   - Every objective has a resolvable expedition parent.
   - No orphan proposals exist.
   - No duplicate proposal identities exist.

4. **Regression tests**
   - Permanent tests verifying parent-reference correctness across all planning outputs.

---

## Acceptance

A snapshot generated by Mission Studio produces an event log where `expedition.missionId` matches an existing `mission.id` and `objective.expeditionId` matches an existing `expedition.id`.

---

## Phases

### Phase 1 — Reproduce the defect

Create a minimal test that demonstrates the current parent-reference mismatch.

### Phase 2 — Fix proposal generation

Update `proposeExpeditions` and `proposeObjectives` to use proposal IDs for parent references.

### Phase 3 — Add graph validation

Introduce a validator that rejects malformed proposal graphs.

### Phase 4 — Regression tests

Add permanent tests that fail if parent references ever break again.

## Risks

| Risk | Mitigation |
|---|---|
| Fix changes snapshot shape | Keep payload schemas unchanged; only fix identity values |
| Existing examples break | Re-run all examples after the fix and update event logs if needed |
| Performance impact | Validation is O(n) over the proposal graph |
