# EXP-DIST-005 — IDE Rules Projection

> **Product expedition.** Generate Cursor, Cline, Windsurf, Roo, Aider, and Continue.dev rules from the Canonical AI Capability Model so coding agents behave correctly inside SYNTH repositories.

**Status:** Proposed  
**Kind:** Product Expedition  
**Priority:** High

## Objective

Produce IDE-specific rules files that tell coding agents how to interact with a SYNTH repository. All rules are projections of the Canonical AI Capability Model and the AI Interaction Manifest.

---

## Origin Evidence

Developers increasingly use AI-powered IDEs. Without rules, these agents may mutate SYNTH repositories incorrectly, bypass governance, or make invalid assumptions. Platform-specific rules provide the right guidance at the right surface.

---

## Required Change

### 1.1 Target platforms

- Cursor (`.cursor/rules`)
- Cline (`.clinerules`)
- Windsurf (`.windsurfrules`)
- Roo (`.roo/rules`)
- Aider (`.aider.conf.yml` or conventions)
- Continue.dev (`.continue/config.json`)

### 1.2 Rule contents

Each rules file should include:

- Repository type and lifecycle phase detection.
- Mutation policy enforcement.
- Required workflow commands.
- Prohibited actions.
- Escalation rules.
- Skill activation triggers.

### 1.3 Generation

Rules are generated by a projection engine that consumes the Canonical AI Capability Model. They are not hand-maintained per platform.

---

## Deliverables

1. Projection engine for IDE rules.
2. Generated rules files for each supported IDE.
3. Documentation on how to install or generate rules.
4. Tests ensuring generated rules respect governance constraints.

---

## Acceptance Criteria

- Rules are generated deterministically from the Capability Model.
- Generated rules cover all supported IDEs.
- Rules do not leak forbidden implementation terminology.
- Rules respect repository-specific mutation policies.

---

## Out of Scope

- Canonical AI Capability Model definition (EXP-DIST-001).
- MCP server (EXP-DIST-003).
- Package distribution (EXP-DIST-004).

---

## Success Criteria

The expedition succeeds when a developer can drop generated rules into their IDE and the agent immediately follows SYNTH workflows without additional instructions.
