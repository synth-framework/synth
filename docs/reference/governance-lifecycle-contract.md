# Governance Lifecycle Contract

This document defines the canonical lifecycle contracts for **Mission** and **Expedition** entities in SYNTH. It is the normative reference for valid states, allowed transitions, emitted events, and mutation classifications.

The runtime implementation of these contracts (event emission, atomicity, recovery) belongs to **EXP-RUNTIME-001 â€” Runtime Correctness**. This document owns the contract itself and the certification that the runtime satisfies it.

## 1. Mission Lifecycle

### 1.1 States

| State | Meaning |
|-------|---------|
| `draft` | A proposal generated by Mission Studio; not yet approved. |
| `active` | Approved and committed to the event log; may host Expeditions. |
| `completed` | All objectives satisfied; no further Expeditions expected. |
| `archived` | Retained for history but no longer active. |

Source of truth: `src/types/state.ts` (`Mission.status`).

### 1.2 Transitions

### 1.3 Events

| Transition | Required Event | Payload Requirements |
|------------|----------------|----------------------|
| Create draft | `MISSION_CREATED` | `payload.mission.id`, `payload.mission.name`, `payload.mission.status == "draft"` |
| Approve | `MISSION_APPROVED` | `payload.id` matches `MISSION_CREATED.payload.mission.id`, `payload.status == "active"` |
| Complete | `MISSION_COMPLETED` | `payload.id` matches mission id, `payload.status == "completed"` |
| Archive | `MISSION_ARCHIVED` | `payload.id` matches mission id, `payload.status == "archived"` |

### 1.4 Invalid Transitions

- `active` â†’ `draft`
- `completed` â†’ `active`
- `archived` â†’ any state

## 2. Expedition Lifecycle

### 2.1 States

| State | Meaning |
|-------|---------|
| `draft` | A proposal generated by Mission Studio; not yet approved. |
| `approved` | Operator has accepted the proposal. |
| `committed` | Approved intent recorded as a runtime entity in the event log. |
| `executing` | Work on the Expedition has begun. |
| `completed` | Expedition produced evidence and was accepted. |
| `cancelled` | Expedition abandoned before completion. |

## 3. Approval Semantics

### 3.1 Required Evidence

Approval requires at least one piece of evidence attached to the draft. Evidence must include:

- `subject`: what the evidence describes.
- `purpose`: why it supports the proposal.
- `confidence`: `low`, `medium`, or `high`.

### 3.2 Confidence Threshold

The default approval confidence threshold is **0.7**. A draft whose overall confidence is below the threshold must be rejected.

### 3.3 Unknown Resolution

Any **blocking** unknown prevents approval regardless of confidence. Non-blocking unknowns must be recorded but do not prevent approval.

### 3.4 Rejection and Revision

When approval is rejected, the CLI must:

- Return a `MissionApprovalDecision` or equivalent with `approved: false`.
- Explain the reason (confidence, unknowns, or missing evidence).
- Suggest the next command to add evidence and create a successor draft.

## 4. Mutation Boundaries

Every public SYNTH command must declare one of the following mutation classes:

| Class | Definition | Allowed in Discovery? | Requires Approval? |
|-------|------------|----------------------:|-------------------:|
| `READ_ONLY` | Observes state without modification | Yes | No |
| `PROPOSAL_ONLY` | Generates proposals or reports without state mutation | Yes | No |
| `POTENTIALLY_MUTATING` | May modify state depending on flags | No | Yes if mutating |
| `MUTATING` | Explicitly modifies repository or governance state | No | Yes |

Discovery phase enforcement: commands outside `READ_ONLY` and `PROPOSAL_ONLY` are rejected with a clear phase-boundary error.

---

## 5. Replay Reconstruction

The canonical state must be fully reconstructible from the event log. For governance entities, replay must produce:

- Mission entities with correct `status` and `expeditions` list.
- Expedition entities with correct `status`, `missionId`, `objectives`, and `discoveries`.
- Consistent identity: no mission id may collide with an expedition id or objective id.

Any divergence between replay output and persisted state is a structural divergence and fails validation.
