# Projection Model

> **Authority:** Architectural Constitution  
> **Status:** Active  
> **Evidence:** `docs/operator/EXP-PROGRAM-010-evidence-annex-taskpro.md` (governed bootstrap E1)

This document defines what is canonical state, what is a projection, and what authority each SYNTH output carries. It resolves the ambiguity that appeared during E1: whether expedition status, repository identity, operator briefings, and generated documentation are sources of truth or derived views.

---

## Source of Truth

The **event log** at `data/event-log.jsonl` is the single source of truth in a SYNTH project. Every other state-bearing artifact is a **projection** of the event log, possibly combined with other replayable evidence (drafts, snapshots, decisions, manifest).

This is a direct consequence of the Constitution:

> **Provision 10.** The event log is the single source of truth. State is a derived projection.

---

## Taxonomy

### Canonical State

**Canonical State** is the authoritative materialized state produced by replaying the event log through domain logic. It is persisted at `data/canonical-state.json` as a performance cache, but its authority derives entirely from the event log. A valid canonical state must reconstruct to the same `stateHash` on replay.

### Projection

A **Projection** is any read-only view computed from events, canonical state, drafts, snapshots, decisions, or the bootstrap manifest. Projections are not authoritative over the event log. They exist to make the system observable, operable, and explainable.

### Cached Projection

A **Cached Projection** is a projection stored on disk for performance, distribution, or tooling. A cached projection is valid only when it carries mandatory provenance linking it to the source state hash and schema version. It must be regenerated when the source changes.

### Source Input Artifact

A **Source Input Artifact** is produced before governance authorization and may become evidence:

- **Mission Drafts** at `data/drafts/*.json` are mutable proposals awaiting approval.
- **Approved Mission Model Snapshots** at `data/snapshots/*.json` are certified, immutable evidence produced by Mission Studio.

Source input artifacts are not projections of the event log, but they are governed inputs to it.

### Bootstrap Artifact

A **Bootstrap Artifact** is produced by `synth init` or `synth bootstrap` to establish the project. The `.synth/manifest.json` and directory scaffolding are bootstrap artifacts. They are generated once and regenerated by `synth init`; they are not a source of truth.

### Forbidden Duplication

**Forbidden Duplication** occurs when the same canonical fact exists in more than one mutable location, or when a projection is treated as editable authority. Examples include:

- Editing `data/canonical-state.json` directly.
- Treating a generated documentation file as a source document.
- Storing confidence in a Mission Draft as a mutable field that approval reads rather than recomputes from evidence.

## Authority Rules

| Rule | Statement |
|---|---|
| R1 | The event log is the only source of truth. |
| R2 | Canonical state is a projection of the event log; its materialized file is a performance cache. |
| R3 | Read surfaces either recompute from replayable evidence or consume a cached projection with valid provenance. |
| R4 | No mutable file may hold a canonical fact. |
| R5 | Source input artifacts are governed; drafts are mutable proposals, snapshots are immutable certified evidence. |
| R6 | Bootstrap artifacts are implementation scaffolding, not governance authority. |
| R7 | Identifiers are allocated from governed state; they are never assumed or duplicated. |

---

## Provenance Requirements

Every cached projection MUST carry at least:

| Field | Meaning |
|---|---|
| `sourceStateHash` | The `stateHash` of the canonical state from which the projection was computed. |
| `computedAt` | Timestamp of computation (wall-clock, used only for human orientation). |
| `schemaVersion` | Version of the projection schema. |
| `generator` | Identifier of the command or capability that produced the projection. |

A cached projection whose `sourceStateHash` does not match the current canonical state MUST be rejected or regenerated.

---

## Invalidation Rules

| Situation | Action |
|---|---|
| `sourceStateHash` of a cached projection does not match current state | Projection is stale; regenerate or reject. |
| Knowledge base files change | `docs/generated/` projections are stale; regenerate. |
| Snapshot hash mismatch on load | Snapshot is tampered or corrupted; reject. |
| Event log is appended | Canonical state cache is stale until next replay-driven reconstruction. |

## E1 Projection Questions Answered

These answers are derived from the artifact classification above and from the field evidence in `docs/operator/EXP-PROGRAM-010-evidence-annex-taskpro.md`.

### Should expedition status live in `.synth/expeditions.json`, or be replay-derived?

**Replay-derived.** Expedition status is part of canonical state (`state.expeditions`). It is computed by replaying `EXPEDITION_CREATED`, `EXPEDITION_APPROVED`, `EXPEDITION_STARTED`, `EXPEDITION_COMPLETED`, and related events. There is no `.synth/expeditions.json` authority.

### Is repository identity a file, a projection, or a decision?

**A recomputed projection.** `synth explain identity` derives repository kind, phase, authority, expected inputs/outputs, and transformation direction from replayable evidence: the manifest, event log, canonical state, drafts, snapshots, and expedition charters. No hand-authored identity file exists.

### Can `synth status` cache its briefing, or must it recompute from events every time?

**Recompute on read.** The operator briefing is a recomputed projection. It reads the manifest, canonical state, event count, and draft directory every invocation. Caching it would create a mutable projection that could diverge from the event log.

### If documentation is a projection, where does the cache live and who invalidates it?

Documentation projections are cached in `docs/generated/`. They are invalidated when the knowledge base under `docs/` changes. The generator skips `docs/generated/` during extraction to avoid re-ingesting derived content. Each generated file SHOULD carry a provenance stamp (`sourceStateHash`, `computedAt`, `schemaVersion`, `generator`) so that consumers and verification tools can detect stale output.

## Governance Invariants

| Invariant | Statement |
|---|---|
| P1 | A projection may never be edited as a source artifact. |
| P2 | A cached projection is valid only if its `sourceStateHash` matches the current canonical state. |
| P3 | No canonical fact may exist in more than one mutable location. |

These invariants are enforced by:

- The `ExecutionGate` as the single mutation authority.
- `synth verify` (EXP-GOV-005) for executable checks.
- The ADR process for any change that moves a concern across the projection boundary.
