## Executive Summary

This document defines the Synth Knowledge Representation (SKR) — the canonical format for engineering knowledge in the Synth platform.

SKR formally separates:
- Canonical engineering knowledge (SKR)
- Planning semantics (Planning Layer)
- Execution semantics (Execution IR)
- Runtime implementation (Event Stream)
- External projections (Projection Adapters)

The previous Synth IR specification mixed implementation concepts (Agents, Tools, Capabilities, Workflows) with canonical engineering concepts. SKR replaces it with a clean separation: **knowledge is canonical, execution is implementation**.

---

## Architectural Principle

> **Engineering knowledge MUST remain independent of execution mechanisms.**

Execution is an implementation of engineering intent. Execution primitives, protocols, vendors, APIs, runtimes, and tooling MUST NOT appear inside the canonical knowledge representation.

## Canonical Vocabulary

### Approved Node Types

SKR recognizes only these canonical node types:

| Node Type | Represents | Example |
|-----------|-----------|---------|
| **Mission** | Long-term strategic direction | "Build deterministic execution kernel" |
| **Expedition** | Bounded engineering objective | "Migrate Ticket → WorkItem" |
| **Objective** | Specific measurable outcome | "Remove Ticket domain wrappers" |
| **WorkItem** | Trackable unit of work | "Implement CanonicalLanguageAuditor" |
| **Discovery** | Newly learned architectural knowledge | "Capability has leaked into 6 layers" |
| **Decision** | Chosen architectural direction | "Capability is bytecode, not architecture" |
| **Artifact** | Document or specification produced | "docs/audits/ASC-001-report.md" |
| **Observation** | Recorded insight or measurement | "110 capability references found" |
| **Constraint** | Limitation or requirement | "Replay must remain deterministic" |
| **Relationship** | Semantic link between nodes | "depends_on", "implements" |

### Approved Relationship Types

| Relationship | Meaning |
|-------------|---------|
| `depends_on` | Target must exist for source to proceed |
| `implements` | Source realizes the intent of target |
| `supports` | Source enables or assists target |
| `derived_from` | Source was created from target |
| `discovers` | Source produced target as knowledge |
| `produces` | Source generates target as output |
| `invalidates` | Source makes target obsolete |
| `blocks` | Source prevents target from proceeding |
| `relates_to` | General semantic association |
| `references` | Source cites or mentions target |

### Forbidden Node Types

The following SHALL NOT appear as canonical SKR node types:

```
Agent, Tool, Workflow, Capability, Runtime, Protocol,
Server, Adapter, Plugin, Transport, Provider, Connector,
MCP, A2A, GitHub, Jira, Linear
```

These belong to execution or projection. They are implementation concerns, not engineering knowledge.

### Forbidden Relationship Types

The following SHALL NOT appear as canonical SKR relationships:

```
invoke, execute, rpc, call, tool_use, http_request,
capability_invoke, workflow_trigger, agent_dispatch
```

These are execution mechanisms. They belong in Execution IR, not in the Knowledge Representation.

## Node Model

Each node represents engineering knowledge. Nodes are protocol-independent.

### Example: WorkItem

```yaml
kind: WorkItem
id: WI-001
title: Introduce Execution Planner
status: planned
metadata:
  createdAt: "2026-06-28T00:00:00Z"
  version: "1.0.0"
extensions: {}  # Implementation-specific; may contain projection data
```

### Example: Discovery

```yaml
kind: Discovery
id: D-001
title: Capability terminology has leaked into 6 architectural layers
confidence: high
observations:
  - "110 capability references found in kernel source"
  - "CapabilityRegistry is an execution concept named as architecture"
related:
  - id: WI-001
    relationship: discovers
```

### Example: Decision

```yaml
kind: Decision
id: DC-001
title: Capability is bytecode, not architecture
status: accepted
alternatives:
  - "Keep Capability as architectural concept"
  - "Demote Capability to execution bytecode"
chosen: 1
consequences:
  positive:
    - "Planning layer becomes cleaner"
    - "Execution can evolve independently"
  negative:
    - "Requires migration of 110 references"
```

### What Nodes Are NOT

Nodes are NOT protocol objects:

```yaml
# WRONG — this is a protocol object, not engineering knowledge
kind: Agent
id: agent-1
role: assistant

# WRONG — this is a tool binding, not canonical knowledge
kind: Tool
id: github-search
api: rest

# WRONG — this is execution choreography
kind: Workflow
id: deploy-pipeline
steps: [...]
```

## Knowledge Graph

The root object is the Synth Knowledge Graph:

```yaml
kind: SynthKnowledgeGraph
version: v1
metadata:
  system: Synth v2
  createdAt: "2026-06-28T00:00:00Z"
nodes: []
relationships: []
```

### Properties

| Property | Type | Description |
|----------|------|-------------|
| `kind` | string | Always `SynthKnowledgeGraph` |
| `version` | string | Schema version (semver) |
| `metadata` | object | System metadata |
| `nodes` | array | All canonical nodes |
| `relationships` | array | All canonical relationships |

### Constraints

- `nodes` MUST contain only approved node types
- `relationships` MUST contain only approved relationship types
- All `id` values MUST be unique within the graph
- `version` MUST follow semantic versioning

## Message Model

Messages become observations, not protocol objects.

### Before (Synth IR)

```yaml
message:
  role: assistant
  content: "I recommend migrating Ticket to WorkItem."
```

### After (SKR)

```yaml
kind: Observation
id: OBS-001
source:
  type: "planning_cognition_engine"
  expedition: "EXP-001"
timestamp: "2026-06-28T00:00:00Z"
content: "I recommend migrating Ticket to WorkItem."
confidence: 0.92
category: "recommendation"
```

Protocol adapters may translate Observations into protocol-specific message formats (OpenAI, Anthropic, MCP, A2A) at the projection layer.

---

## Extensions

Extensions are permitted but ONLY for implementation-specific information.

### Example

```yaml
kind: WorkItem
id: WI-001
title: Introduce Execution Planner
extensions:
  github:
    issue_number: 42
    labels: ["enhancement", "architecture"]
  jira:
    ticket_id: "ARCH-123"
  mcp:
    resource_uri: "synth://work-items/WI-001"
```

**Rule:** Core SKR MUST remain protocol-independent. Extensions contain projection-specific data only.

## Serialization Rules

SKR maintains these properties:

| Property | Guarantee |
|----------|-----------|
| **Deterministic** | Same input always produces same output |
| **Canonical** | Single source of truth for engineering knowledge |
| **Stable** | Schema evolves through ADRs, not ad-hoc changes |
| **Versioned** | Every document has a schema version |
| **Replayable** | Knowledge can be reconstructed from events |
| **Serializable** | YAML, JSON, and binary formats supported |

### Stable Ordering

- Nodes are sorted by `kind`, then `id`
- Relationships are sorted by `source`, then `target`, then `type`
- Object keys are sorted alphabetically

### Stable Identifiers

- IDs are assigned at creation and never change
- ID format: `{KIND_PREFIX}-{SEQUENCE}` (e.g., `WI-001`, `EXP-042`)

## Execution IR

Execution IR is NOT SKR. It is a separate artifact.

### Relationship

```
Decision
  ↓
Execution Plan
  ↓
Execution IR          ← Compiler output from planning
  ↓
Execution Primitive Sequence
  ↓
Capability Invocation
  ↓
Event Stream
```

Execution IR is the compiler output generated from planning. It contains:
- Capability sequences
- Runtime bindings
- Protocol mappings
- Vendor-specific configurations

SKR contains NONE of these.

### Example: Execution IR

```yaml
# This is Execution IR, NOT SKR
kind: ExecutionPlan
id: EP-001
decision: DC-001
steps:
  - capability: CreateWorkItem
    params: { id: "WI-001", title: "Introduce Execution Planner" }
  - capability: CreateExpedition
    params: { id: "EXP-001", goal: "Migrate to SKR" }
```

This document lives in the Execution Layer, not the Knowledge Layer.

## Planning Layer

Planning operates exclusively on canonical engineering concepts.

```
Intent
  ↓
Understanding
  ↓
Planning
  ↓
Decision
  ↓
ExecutionPlan       ← Still canonical (planning artifact)
```

Execution instructions SHALL NOT appear in planning artifacts. An `ExecutionPlan` in the Planning Layer is a sequence of canonical operations ("Create WorkItem", "Chart Expedition"), not capability invocations.

---

## Execution Layer

### Vocabulary

| Current | Target | Rationale |
|---------|--------|-----------|
| `CapabilityRegistry` | `ExecutionPrimitiveRegistry` | Capabilities are execution primitives |
| `Capability` | `ExecutionPrimitive` | Or "ExecutionInstruction" / "ExecutionOperation" |
| `handleIntent()` | `executePlan()` | Intent is handled at API; execution is planned |

### Invariants

1. Execution primitives may evolve without changing planning semantics.
2. New primitives can be added without SKR changes.
3. Primitive parameters are implementation details.

---

## Projection Layer

Projection adapters are the ONLY layer aware of external systems.

### Adapters

```
GitHubAdapter       → GitHub Issues, PRs, Projects
JiraAdapter         → Jira Tickets, Epics, Sprints
LinearAdapter       → Linear Issues, Cycles
FilesystemAdapter   → Local files, Markdown
MCPAdapter          → Model Context Protocol resources
A2AAdapter          → Agent-to-Agent protocol
SlackAdapter        → Slack notifications
```

### Invariant

No external vocabulary may leak upward into canonical knowledge. A WorkItem in SKR is always a WorkItem. The GitHubAdapter may project it to a GitHub Issue, but the SKR node remains protocol-independent.

## Guiding Principle

> **Knowledge is canonical. Planning reasons about knowledge. Execution realizes plans. Projection communicates results.**

Every architectural layer speaks only its own ubiquitous language, with explicit translation boundaries between layers. This separation is a permanent architectural invariant and SHALL guide all future evolution of the Synth platform.
