# Continuous Publication

SYNTH treats every merge to `main` as a publication event.

## Pipeline

```text
Pull Request
     ↓
Proof Gate  (npm run govern)
     ↓
Merge to main
     ↓
Publish workflow
     ├── Regenerate docs/generated/
     ├── Commit regenerated docs
     └── Deploy website/ to GitHub Pages
     ↓
Tag v*.*.*
     ↓
Release workflow
     ├── Run npm run govern
     └── Create release with proof artifact
```

No manual deployment step is required.

## Workflows

| File | Trigger | Purpose |
|---|---|---|
| `.github/workflows/proof.yml` | Push/PR to `main` | Canonical verification gate. Runs `npm run govern` and uploads the proof artifact. |
| `.github/workflows/publish.yml` | Push to `main` | Regenerates public documentation and deploys the static website. |
| `.github/workflows/release.yml` | Push tag `v*.*.*` | Creates a semantic release and attaches the proof artifact. |

## Documentation regeneration

The `publish.yml` workflow runs:

```bash
npm run build
npm run docs:generate
```

This projects the knowledge base in `docs/` into `docs/generated/`. If the generated files change, the workflow commits them back to `main` with the message `docs: regenerate documentation [skip ci]`.

## Website deployment

The `website/` directory contains static HTML and CSS. The workflow uploads it as a GitHub Pages artifact and deploys it to the configured Pages environment.

## Release artifacts

When a maintainer pushes a semantic-versioned tag such as `v2.1.0`, the release workflow:

1. Runs `npm run govern`.
2. Creates a GitHub release.
3. Attaches the generated `proof/proof-*.json` artifact.
4. Generates release notes from the changelog.

## Determinism guarantee

Every published state of SYNTH is replayable from Git history because:

- `main` can only be updated through reviewed, passing pull requests.
- The proof artifact is generated by `npm run govern` on every build.
- The same proof artifact is attached to every release.
