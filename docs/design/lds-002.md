# Mission Studio Design System (LDS-002)

> **Mission Studio is the primitive of SYNTH's public interface.** The SYNTH homepage is one projection of Mission Studio. Future surfaces — desktop, VS Code extension, mobile, documentation previews — will consume the same runtime contract and render through this design system.

---

## Thesis

Do not design a homepage and then add a design system. Do not design components and then assemble a page. Design **Mission Studio** first. The homepage, the Storybook, and every future surface are projections of that single application.

This system is organized from the largest structure to the smallest primitive:

```text
Mission Studio
  ↓
Workspace
  ↓
Artifacts
  ↓
Interactions
  ↓
Primitives
```

Buttons, inputs, and badges are derived elements. They exist to serve Mission Studio, not to define it.

---

## 1. Mission Studio

Mission Studio is the canonical SYNTH application surface. It is persistent, state-machine-driven, and artifact-centered.

### 1.1 Core structure

```text
┌─────────────────────────────────────────┐
│ Header                                  │
├──────────┬──────────────────────────────┤
│          │                              │
│ Sidebar  │   Workspace                  │
│          │   (Artifact Surface)         │
│          │                              │
├──────────┴──────────────────────────────┤
│ Footer / Status                         │
└─────────────────────────────────────────┘
```

### 1.2 Public principles

1. **Runtime first.** Every visible element corresponds to a runtime object.
2. **Workspace over pages.** Everything lives inside one workspace; scrolling changes workspace state.
3. **Progressive disclosure.** Information density increases as the lifecycle progresses.
4. **Artifact driven.** Replace chat with objects. Everything shown is an artifact.
5. **Calm computing.** Large whitespace, low visual noise, subtle motion, clear hierarchy.
6. **Themeable by contract.** Every surface derives from tokens; themes are data, not overrides.

---

## 2. Workspace

The Workspace is the persistent shell that contains the active phase and its artifacts.

### 2.1 Header

- **Identity:** Mission Studio wordmark / logo.
- **Mission title:** current demo mission name.
- **Phase badge:** current lifecycle phase.
- **Governance status:** governed / draft.
- **Global actions:** theme toggle, documentation link.

### 2.2 Sidebar

The Sidebar is **lifecycle progress**, not navigation.

```text
✓ Intent
● Discovery
○ Constraints
○ Domain
○ Mission
○ Expeditions
○ Governance
○ Replay
○ Architecture
○ Repository
```

- Completed phases show a checkmark.
- Active phase shows a filled indicator.
- Pending phases show an empty indicator.
- Phase labels are calm and readable.

### 2.3 Artifact Surface

The main area where artifacts appear. It is generous, readable, and scrollable.

### 2.4 Footer / Status

- Runtime status message.
- Evidence / replay position / repository summary.
- State hash during Replay.

---

## 3. Artifacts

Artifacts are the central objects of Mission Studio. They should feel closer to documents than widgets.

### 3.1 Artifact composition

Every artifact contains, in order:

1. **Kind label** — small, uppercase, muted.
2. **Title** — large, scannable.
3. **Status / confidence** — prominent metadata.
4. **Body** — findings, objectives, observations.
5. **Grouped metadata** — entities, constraints, sources.
6. **Semantic surface** — full background color, not just a border.

### 3.2 Artifact example

Instead of:

```text
┌──────────────┐
│ DISCOVERY    │
│ domain-model │
│ auth         │
└──────────────┘
```

Use:

```text
┌──────────────────────────────────────┐
│ DISCOVERY                            │
│                                      │
│ Intent analyzed                      │
│                                      │
│ Confidence        96%                │
│ Unknowns          3                  │
│ Findings          12                 │
│                                      │
│ ───────────────────────────────────  │
│ Detected                             │
│ • Architecture                       │
│ • Constraints                        │
│ • Dependencies                       │
│                                      │
│ Ready for Mission                    │
└──────────────────────────────────────┘
```

### 3.3 Artifact kinds

| Kind | Semantic color | Surface role |
|---|---|---|
| Intent | Genesis blue | Origin / creation |
| Discovery | Replay cyan | Understanding / analysis |
| Unknowns | Knowledge red | Open questions |
| Domain | Governance purple | Model / boundaries |
| Mission | Mission blue | Purpose / commitment |
| Expedition | Expedition amber | Work / execution |
| Evidence | Evidence gold | Observation / confidence |
| Governance | Governance purple | Approval / boundary |
| Replay | Replay cyan | History / determinism |
| Architecture | Architecture emerald | Layered system |
| Repository | Repository teal | Source / state |

## 4. Interactions

### 4.1 Scroll-driven phase transitions

Page scroll maps to Mission Studio phase. The workspace remains pinned; only internal state changes.

### 4.2 Phase transitions

- Use crossfade + subtle vertical translation.
- Artifact cards enter with staggered delay.
- Sidebar indicator animates to the new phase.
- Reduced motion collapses transitions.

### 4.3 Replay scrub

- Timeline is horizontal or integrated into the footer.
- Scrubbing updates artifacts to reflect historical state.
- State hash updates visibly.

---

## 5. Primitives

Primitives are derived from the layers above. They should never define the design system.

### 5.1 Buttons

- Used sparingly.
- Primary action: filled accent.
- Secondary action: subtle border.
- Icon buttons: circular, minimal.

### 5.2 Inputs

- Large, calm, generous padding.
- Clear focus state.
- Placeholder text is helpful, not noisy.

### 5.3 Badges / status

- Integrated into artifact metadata.
- Color matches semantic concept.
- Uppercase, small, tracked.

## 6. Tokens

Tokens are the machine-readable foundation. They are organized by domain and applied through the hierarchy above.

| `--ms-surface-2` | `#F1F3F5` | `#1f2128` | Subtle elevated surfaces |

| `--ms-border-strong` | `#CED4DA` | `#3c404d` | Focused borders |

## 7. Layout

### 7.1 Desktop (>1024px)

- Full three-panel workspace: sidebar, artifact surface, footer.
- Artifact surface uses generous padding.
- Cards can be full-width or two-column depending on content.

### 7.2 Tablet (768–1024px)

- Sidebar collapses to a compact horizontal stepper or remains as a rail.
- Workspace remains sticky.

### 7.3 Mobile (<768px)

- Sidebar becomes a horizontal phase strip or hideable drawer.
- Workspace stacks vertically.
- Status bar remains visible.
- Artifacts remain document-like, not compressed into widgets.

---

## 8. Accessibility

- Skip link to main content.
- Focus-visible styles on all interactive elements.
- Live region announces phase changes.
- Keyboard navigation through controls.
- `prefers-reduced-motion` disables phase transitions.
- WCAG 2.2 AA contrast targets.

---

## 9. Design Principles Checklist

Use this checklist when reviewing any Mission Studio surface:

- [ ] The element maps to a documented SYNTH concept.
- [ ] It uses only tokens defined in this document.
- [ ] It does not introduce arbitrary colors outside the semantic palette.
- [ ] Motion, if present, uses documented duration/easing tokens.
- [ ] It respects the workspace metaphor (no page-like transitions).
- [ ] It supports progressive disclosure (default state is calm; detail is reachable).
- [ ] It includes a reduced-motion fallback if it animates.
- [ ] Copy uses SYNTH public vocabulary only.
- [ ] It is traceable to an expedition charter or runtime artifact.
- [ ] It is evaluated in the context of the full Mission Studio workspace, not in isolation.

---

## 10. Conformance

Any deviation from this system must be documented in the expedition charter and accepted as part of the expedition review. The Storybook is the primary conformance artifact: if a component cannot be shown in a Mission Studio workspace state, it does not belong in the public interface.
