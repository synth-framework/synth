# First Contact Experience v2 Specification

## Purpose

This specification defines how an autonomous reasoning entity should enter a SYNTH-governed repository, construct a correct initial model, and act through governed workflows rather than implementation assumptions.

First Contact v2 treats the entry experience as a **semantic deformation problem**: the agent begins with an unknown interpretation landscape, and SYNTH must supply the minimum evidence required to collapse that landscape into the correct attractor state before execution begins.

---

## Principles

### 1. Evidence before action

An agent must inspect the repository through SYNTH commands before proposing changes. The first action is never file modification.

### 2. Intent fidelity over task completion

Success is measured by how well the agent reconstructs the human intent, not by how quickly it produces code.

### 3. Governed change only

Capability requests are transformed into missions and expeditions. Implementation is downstream of approval.

### 4. Uncertainty is preserved

Ambiguous requests must be returned as questions, not compressed into assumed intent.

### 5. Patterns are derived, not authored

Quick-start conversation patterns come from observed successful trajectories, not from intuition.

---

## Entry contract

Every agent entering a SYNTH repository agrees to the following behavior contract.

### Before acting, the agent shall

1. Run `synth status` to establish current lifecycle phase and blockers.
2. Run `synth explain` to read the operator briefing and canonical state.
3. Run `synth docs generate` to materialize the knowledge projections if needed.
4. Run `synth mission list` and `synth expedition list` to understand active work.

### The agent shall not

1. Modify implementation files before a mission is approved.
2. Treat missing files as errors without checking repository intent.
3. Generate code for ambiguous or ungoverned requests.
4. Bypass `synth mission create`, `synth expedition create`, or `synth mission approve`.

### On ambiguous requests, the agent shall

1. State that the request is unclear.
2. Identify missing acceptance criteria, user impact, and current limitations.
3. Offer to create a mission draft once intent is clarified.

## The learning loop

First Contact v2 is a closed learning system:

```text
Observed agent session
        |
        v
FirstContactEvidence
        |
        v
ConversationPattern (provisional)
        |
        v
Pattern validation
        |
        v
ConversationPattern (canonical)
        |
        v
Quick-start projection
        |
        v
Future agent onboarding
        |
        v
Observed agent session
```

Each iteration improves the entry experience without changing SYNTH architecture.

## Artifact model

### `FirstContactEvidence`

Captures one observed first-contact session.

- `repositoryState`
- `agentInitialModel`
- `evidenceConsumed`
- `interpretationChanges`
- `interventions`
- `finalModel`
- `successConditions`

Stored in `first-contact/sessions/*-evidence.json`.

### `ConversationPattern`

A reusable, evidence-backed trajectory for a common first-contact scenario.

- `id`
- `title`
- `trigger`
- `preconditions`
- `trajectory`
- `decisionPoints`
- `successfulPrompts`
- `antiPatterns`
- `misinterpretationCategories`
- `confidence`
- `status` (`provisional`, `canonical`, or `rejected`)
- `supportingEvidence`

Stored in `first-contact/conversation-patterns/*-pattern.json`.

### Quick-start projection

A Markdown document derived from a canonical `ConversationPattern` for human operators and agents.

Stored in `docs/first-contact/quick-start/*.md`.

Regenerated by `node scripts/generate-first-contact-quickstart.js`.

## Misinterpretation taxonomy

| Category | Example | Corrected interpretation |
|---|---|---|
| Intent confusion | "Is this an application?" | "This is a specification repository." |
| Missing context | "No package.json means the project is broken." | "The project is in a pre-implementation state." |
| Governance confusion | Modifying `.synth/` directly | Working on the product through governed expeditions |
| Vocabulary mismatch | "Mission" interpreted as "task list" | "Mission" as governed transformation objective |

---

## Validation gates

A `ConversationPattern` is promoted from `provisional` to `canonical` only when:

1. It has at least one supporting evidence artifact.
2. Its confidence is at or above the repository threshold.
3. Extraction is deterministic across repeated runs.
4. No conflicting trajectory exists for the same trigger and preconditions.

Patterns that fail validation remain `provisional` or are marked `rejected`.

---

## Onboarding contract for agent implementations

Agents that interact with SYNTH repositories should:

1. Read this specification before their first session.
2. Begin every session with the entry contract sequence.
3. Use canonical quick-start projections as reference trajectories.
4. Contribute new `FirstContactEvidence` when they encounter unclassified scenarios.
5. Never treat expeditions as task lists or missions as feature requests.

## Success metrics

### Semantic alignment score

| Dimension | Question |
|---|---|
| Intent recognition | Did the agent understand why the request was made? |
| Boundary recognition | Did the agent distinguish SYNTH from the project? |
| Evidence seeking | Did the agent inspect before acting? |
| Governance compliance | Did the agent use approved paths? |
| Recovery ability | Did new evidence correct a wrong model? |
| Compression efficiency | How little context was required to reach understanding? |

This specification governs the **entry experience** only. It does not modify:

- SYNTH architecture
- CLI command semantics
- Governance model
- Event model
- Protected Assets
